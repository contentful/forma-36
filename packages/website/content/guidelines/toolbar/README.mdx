## Toolbar

---

### Purpose

The toolbar pattern provides global and consistent access to critical tools and contextually relevant apps within the platform. It addresses user needs for quick actions, enhances efficiency by minimizing navigation friction, and aligns closely with user workflows. By providing focused and immediate interactions, the toolbar pattern supports a streamlined editing experience, integrates seamlessly with different platform views, and reinforces the user's core jobs-to-be-done across various tasks.

---

### When to Use

Use the toolbar pattern when users frequently need quick, persistent access to tools and functionalities directly related to their primary tasks, enabling them to perform actions without leaving their current context.

#### Typical Use Cases

- Engaging agents for tasks like spinning up new experiences, viewing analytics insights, or performing other specialized tasks through a chat interface.
- Quick access to optimization tools relevant to content performance and management.
- Collaborative workflows that require persistent commenting and feedback mechanisms.
- Seamless integration and access to third-party apps directly within the editing context.
- Workflow management activities that benefit from immediate and persistent tool accessibility.

---

### Use a Different Pattern Instead

Consider other patterns when persistent access to tools is unnecessary or when providing extensive tool sets might overwhelm the user.

| Scenario                                                       | Alternative Pattern | Reason                                                                        |
|----------------------------------------------------------------|---------------------|-------------------------------------------------------------------------------|
| Simple actions within a limited context                        | Action Buttons      | A toolbar would add unnecessary complexity                                    |                 |
| Single task-focused tools without ongoing context              | Modal Dialog        | Ensures users remain focused without permanent interface elements             |
| One-time setup or configuration tasks                          | Wizard              | Provides step-by-step guidance without persistent toolbar complexity          |
| Primary actions that should always be prominently available    | Header Actions      | Ensures immediate visibility for critical actions                             |
| Essential information and actions visible at all times         | Persistent Sidebar  | Offers constant visibility within content layout                              |
| Long forms and extensive editing tasks                         | Dedicated Page      | Provides appropriate space and focus for extensive content manipulation       |

### Typical Interactions

#### Example Flow

1. User selects an icon in the toolbar.
2. The corresponding panel expands.
3. The user performs their required action within the panel (e.g., viewing insights, commenting, or task management).
4. Once completed, the user can collapse the panel via the close button in the panel header or by clicking the corresponding icon in the toolbar again.
5. Alternatively, the user can switch directly to another toolbar item as needed.

### States


#### Toolbar States

- **Default:** Toolbar tabs are available for selection.
- **Loading:** Tab or toolbar data is loading.

#### Toolbar tab States

- **Default:** Tabs are idle and available.
- **Hover:** Tabs visually indicate interactivity when hovered.
- **Selected:** Tabs clearly highlight when their corresponding panel is active.
- **Error:** Indicates issues loading Toolbar tabs or functionality.

#### Panel States

- **Expanded:** Panel is fully visible and interactive.
- **Collapsed:** Panel is minimized, but accessible via the toolbar icon.
- **Loading:** Panel content is loading. Refer to the [Loading](insert link) pattern.
- **Error:** Content inside the panel failed to load or perform actions. Refer to the [Error Handling](insert link) pattern.

### Accessibility (W.I.P.)

#### Screen Reader Support
##### Aria roles
| Element             | Role/Attribute                | Purpose                             |
|---------------------|-------------------------------|-------------------------------------|
| Container for icons | role="tablist"                | Groups the tab controls             |
| Each icon button    | role="tab"                    | Identifies each control             |
| Each panel          | role="tabpanel"               | The content area                    |
| Tablist             | aria-orientation="vertical"   | Critical for your vertical layout   |
| Active tab          | aria-selected="true"          | Shows which tab is currently active |
| Tab → Panel link    | aria-controls="panel-id"      | Associates tab with its panel       |
| Panel → Tab link    | aria-labelledby="tab-id"      | Labels the panel by its tab         |

#### Keyboard Interactions

- **Up/Down Arrow:** Navigate between tabs (not Left/Right)
- **Enter/Space:** Activate the focused tab (optional—can auto-activate on focus)
- **Esc:** Collapse an expanded panel.
- **Tab:** Move focus into the panel content
- **Shift+Tab:** Return focus to the active tab

#### Landmarks
- Use a single region landmark for the entire panel with a clear label
- Use headings (h2, h3, etc.) to structure content within the panel rather than nested landmarks
- Reserve additional landmarks only if the panel has truly distinct, complex sections

#### Announcing changes
- Use live regions to announce the opening and closing of the panel and witch panel opened
- Updates on background processes that happen insde tha panel, open or closed. For example: an Agent has completed a run.

#### Cognitive Considerations

- Consistently place Toolbar tabs to foster intuitive use and reduce learning curves
- Use clear and concise labels to facilitate quick recognition of each action

### Dos and Don'ts

#### Dos

- Show loading and error states for both the tab strip and panel content; defer to the shared Loading and Error Handling patterns
- Trigger panel opening only from buttons, never from links

#### Don'ts

- Add actions to the Toolbar tabs. The Toolbar tabs are a navigational device an functionality should only be shwon inside the open panel.
- Switch the order of tabs between different views
- Open or close the panel without a deliberate user interaction
